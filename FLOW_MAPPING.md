# Data Flow and MBA Execution Audit Mapping

This document audits and maps the complete lifecycle of a variable from initialisation through compile-time transformation, serialisation, transmission scrambling, VM extraction, and stack-based execution. It traces a variable `x` initialised via a complex polynomial Mixed Boolean-Arithmetic (MBA) addition: `let x = a + b;`.

---

## 1. Initialisation (Source Code)
The variable lifecycle begins in the source program with a local variable declaration and assignment representing integer addition:
```javascript
let x = a + b;
```
Here, `a` and `b` are previously declared integer variables stored in local memory slots.

---

## 2. Compiler AST (Abstract Syntax Tree)
The parser in [parser.ts](file:///Users/luke/Desktop/fortress-wasm/compiler/src/parser.ts) structures the source tokens into a strongly-typed AST:
- **LetStatement**:
  - `type`: `"LetStatement"`
  - `name`: `Identifier { name: "x" }`
  - `value`: `BinaryExpression`
    - `type`: `"BinaryExpression"`
    - `operator`: `"+"`
    - `left`: `Identifier { name: "a" }`
    - `right`: `Identifier { name: "b" }`

---

## 3. Bytecode Generation and Polynomial MBA Substitution
When the `CodeGenerator` in [codegen.ts](file:///Users/luke/Desktop/fortress-wasm/compiler/src/codegen.ts) visits the `LetStatement` and its `BinaryExpression` values:
1. It checks if the operands are integers and if `DEV_MODE` is disabled (`process.env.DEV_MODE !== 'true'`).
2. If so, it replaces the simple addition with a **Polynomial MBA Identity**:
   $$a + b = (a \oplus b) + 2 \cdot (a \land b) + 0$$
    To expand the mathematical domain and defeat automated deobfuscators, it introduces a randomised dummy variable `z` (already initialised in the local frame) and emits the equivalent polynomial sequence:
    $$a + b = (a \oplus b) + ((a \land b) \ll 1) + ((z^{2} + z) \land 1) \cdot a$$
    Note: Because z² + z = z(z + 1) is always even for any integer z, the term (z² + z) & 1 (bitwise AND) is always 0, making the final term 0 · a = 0.
3. The compiler allocates local stack slots for temporary variables `_mba_temp_r` and `_mba_temp_l` to hold operands on the stack.
4. The instructions are translated into canonical opcode values (e.g., `LoadLocal`, `BitXor`, `BitAnd`, `Shl`, `Add`, `Mul`, `StoreLocal`).
5. **Instruction Scrambling**:
   The compiler generates a randomised `opcodeMap` mapping canonical opcodes to randomised byte values. For example, if `OpCode.Add` is mapped to `0xD4`, then `0xD4` is written to the bytecode array instead of `OpCode.Add`'s raw enum value.

---

## 4. EAKE, Scrambler Encryption & Decoupled Telemetry Carrier
The server-side scrambler in [scrambler.ts](file:///Users/luke/Desktop/fortress-wasm/server/scrambler.ts) processes the generated bytecode:
1. **Key Negotiation & Signature**: Derives an Ed25519 signing key using Argon2id from `FORTRESS_SIGNING_PASSWORD` and salt. Generates an ephemeral X25519 DH keypair, computes a 32-byte session key via DH exchange with the client, and signs the handshake header (X25519 public key and timestamp) via Ed25519.
2. **Replay Check**: Nonces are checked against the server NonceStore to enforce replay protection within a 5-minute replay window.
3. **Payload Encryption**: The canonical `.fvbc` payload is encrypted with a rolling 32-byte XOR key derived from the negotiated session key.
4. **Decoupled Telemetry Carrier**: The legacy steganography mechanism is decoupled from VM execution key negotiation. It is now used solely to embed the telemetry signature verification key within `logo.png`.

---

## 5. VM Decryption & Verification
When the client-side WebAssembly runtime executes the obfuscated payload:
1. **Handshake Verification**: The VM performs constant-time validation of the handshake header's Ed25519 signature and timestamp using the `subtle` crate.
2. **Session Key Derivation**: The VM derives the 32-byte session key and `base_key_material` using X25519 DH and HKDF-SHA256. Once the keys are derived, the client's ephemeral private key is immediately zeroized in memory.
3. **VirtSC Integrity Check**: The VM computes the keyed HMAC-SHA256 of the scrambled payload using `base_key_material` and compares it to the expected HMAC. If a mismatch is detected, the VM zeroizes all sensitive memory and aborts execution.
4. **JIT Sliding Decryption & Page Hash Check**: The VM JIT-decrypts the bytecode page-by-page (in 256-byte chunks) using the rolling session key. It computes the SHA-256 hash of each page and compares it to the expected page hash. If a hash mismatch is detected (dynamic tampering), the VM XORs `session_key[0]` with `0xFF` to silently corrupt subsequent decryptions.
5. **Zeroization**: Upon completion or verification failure, all sensitive key materials and memory regions are explicitly wiped using `zeroize`.

## 6. VM Stack Execution Audit
The VM interpreter in [vm.rs](file:///Users/luke/Desktop/fortress-wasm/crates/vm-core/src/vm.rs) executes the bytecode by decoding each randomised byte and manipulating the evaluation stack. The step-by-step stack state during the execution of the polynomial MBA addition is audited below:

| Step | Canonical Opcode | Operand(s) | Stack State (Bottom -> Top) | Description |
|---|---|---|---|---|
| 1 | `LoadLocal` | `slot(a)` | `[a]` | Load value of `a` |
| 2 | `LoadLocal` | `slot(b)` | `[a, b]` | Load value of `b` |
| 3 | `BitXor` | | `[(a ^ b)]` | Bitwise XOR of top two values |
| 4 | `LoadLocal` | `slot(a)` | `[(a ^ b), a]` | Load `a` |
| 5 | `LoadLocal` | `slot(b)` | `[(a ^ b), a, b]` | Load `b` |
| 6 | `BitAnd` | | `[(a ^ b), (a & b)]` | Bitwise AND |
| 7 | `PushInt` | `1` | `[(a ^ b), (a & b), 1]` | Push immediate `1` |
| 8 | `Shl` | | `[(a ^ b), ((a & b) << 1)]` | Shift left |
| 9 | `Add` | | `[(a ^ b) + ((a & b) << 1)]` | Add XOR and shifted-AND terms |
| 10 | `LoadLocal` | `slot(z)` | `[temp1, z]` | Load dummy variable `z` |
| 11 | `Dup` | | `[temp1, z, z]` | Duplicate `z` |
| 12 | `LoadLocal` | `slot(z)` | `[temp1, z, z, z]` | Load `z` again |
| 13 | `Mul` | | `[temp1, z, (z * z)]` | Multiply `z * z` |
| 14 | `Add` | | `[temp1, (z * z + z)]` | Add to get polynomial term |
| 15 | `PushInt` | `1` | `[temp1, (z * z + z), 1]` | Push immediate `1` |
| 16 | `BitAnd` | | `[temp1, ((z * z + z) & 1)]` | Bitwise AND (resolves to `0`) |
| 17 | `LoadLocal` | `slot(a)` | `[temp1, 0, a]` | Load `a` |
| 18 | `Mul` | | `[temp1, (0 * a)]` | Multiply (resolves to `0`) |
| 19 | `Add` | | `[temp1 + 0]` | Add dummy term to main result |
| 20 | `StoreLocal` | `slot(x)` | `[]` | Pop final result and store in local variable `x` |

This stack sequence preserves the exact mathematical value of `a + b` while completely hiding the operation inside a complex, non-linear bitwise and polynomial instruction sequence.
